# FairValue Engine — 복합금융상품·파생상품 Pricing 플랫폼 기획서

> **문서 성격**: PRD(제품요구정의서)와 기능명세서의 중간 수준 설계 문서
> **버전**: v1.0 (1차 상용 버전 기준 / Full Pricing Platform)
> **작성 관점**: 금융공학·회계기준·평가실무·웹서비스 기획·백엔드 아키텍처 통합 설계

---

## 0. 지식재산권 및 설계 독립성 고지 (Scope & IP Notice)

이 문서는 시장에 존재하는 상용 복합금융상품 평가 솔루션의 **기능적 범위(functional scope)** 를 벤치마킹하여, 독립적으로 재설계한 평가 플랫폼의 설계서이다. 다음을 전제로 한다.

- 본 설계의 평가엔진은 **공개된 학술 금융공학 모형**(Black–Scholes–Merton, 이항/삼항 격자, Tsiveriotis–Fernandes, Hull–White, Monte Carlo/LSMC 등)과 **공개된 회계기준**(K-IFRS 1032/1109/1102, IFRS 13)에만 근거한다. 특정 사업자의 비공개 알고리즘·소스코드·내부 파라미터는 사용하지 않는다.
- 화면 구성·메뉴 배치·색상·문구·브랜드는 본 문서에서 **독립적으로 재정의**한다. 벤치마킹 대상의 표현적 요소를 복제하지 않는다.
- 금융상품의 평가 **워크플로우 자체**(계약조건 입력 → 권리조건 입력 → 커브 매핑 → 계산 → 결과/보고서)는 IFRS 공정가치 평가 실무에서 도메인이 강제하는 절차이며, 특정 사업자의 창작적 표현이 아니다.

> 실제 구현 전, 동종 등록특허가 있을 경우 **KIPRIS에서 청구범위(claims)를 직접 확인**하고, 청구항 구성요소를 그대로 구현하지 않는 방향으로 설계·검토(필요 시 변리사 침해감정)할 것을 권고한다. 본 문서는 법률자문이 아니다.

## 1. 첨부 화면 분석 요약

벤치마킹 화면에서 추출한 정보를 **A. 직접 확인 / B. 합리적 추론 / C. 신규 제안** 으로 구분한다. 화면을 그대로 모사하지 않고 *기능적 의미*만 추출했다.

### 1.1 A. 화면에서 직접 확인되는 기능

| 영역 | 직접 확인된 요소 |
|---|---|
| 상품군(IA) | RCPS, CPS, CB, EB, BW, Stock Option, 조건부 Stock Option, 등급커브 업로드 / 별도 영역에 Bootstrapping(Parameter), 내부통제(MRC), Overview, Support |
| 평가기준 설정(Step1) | 평가기준일, 분류(예: "회사채-사모무보증"), 적용신용등급(예: B-), **평가기준일에 따른 분류·신용등급 자동 매핑** |

| 채권정보(Step2) | 발행일, 만기일, 발행금액, 액면/발행가, 표면이자율(%), 지급주기(M), 보장수익률(%) |

| 상환권정보(Step4) | 행사 시작일/종료일, 행사가(%), 행사형태(American/European/Bermudan = "STEP MONTH") |
| 매도청구권정보 | 할인유형(예: "독립파생상품-무위험"), 행사 시작일/종료일, 행사가(%), 행사비율(%), 행사형태 |
| 계산결과(Step5) | (CB 예) 채권/상환권/전환권/**매도청구권(음수 가능)**/합계, Parameter: Risk Free Rate, YTM, Parity / (RCPS 예) 우선주/상환권/전환권/매도청구권/합계 |
| 액션 버튼 | 불러오기, 계산, 결과저장, 계산검증 데이터요청, 검증요청, 결과파일(Excel), 보고서(PDF), 임시저장, 임시저장 불러오기, 초기화, 링크복사 |
| 커브 업로드 | 권한요청 → 관리자 승인 → 업로드 활성화, 기준일자 선택, 원본 파일 업로드, 소속 회사 사용자 간 공유, 기준일별 업로드 이력, 등급별 만기(3M~50Y) Rate 테이블 |

**해석 포인트(직접 확인 기반의 도메인 사실):**
- 매도청구권이 결과에서 **음수**로 표시(예: -157,041.84)된다는 것은, 발행자/제3자가 보유한 콜 성격의 권리가 **투자자(보유자) 가치를 차감**함을 의미한다. → 권리 보유 주체별 부호 규약이 엔진에 필요.
- "할인유형: 독립파생상품-무위험"은 매도청구권을 **내재파생이 아닌 별도 파생자산으로 분리해 무위험할인**하는 회계처리 옵션이 존재함을 의미한다.
- 분류·신용등급 자동 매핑은 **평가기준일 기준 커브 스냅샷 + 등급 매핑 테이블**이 백엔드에 존재함을 의미한다.

## 2. 서비스 개요

**FairValue Engine**은 복합금융상품(RCPS·CPS·CB·EB·BW)과 주식보상(Stock Option·조건부 Stock Option)의 **공정가치를 자동 산출하고, 계산근거를 추적 가능하게 보관·문서화**하는 웹 기반 B2B 평가 플랫폼이다.

핵심 명제는 세 가지다.

1. **표준화된 입력**: 상품별 계약조건과 권리조건을 구조화된 폼으로 입력 → 엑셀 수기 평가의 재현성·검증성 문제 해결.
2. **검증 가능한 엔진**: 모형·파라미터·중간 산출물·결과를 스냅샷으로 저장 → 감사 대응 가능한 평가근거 자동 생성.
3. **통합 워크플로우**: 평가기준일·커브 매핑 → 입력 → 검증 → 계산 → 민감도/시나리오 → 보고서까지 one-stop.

이 문서에서 **내부통제(MRC) 기능은 범위에서 제외**하고, Pricing / Parameter / Curve Management / Valuation Result / Report / Audit Trail(계산근거 관리)에 집중한다.

## 3. 문제 정의 및 기획 배경

| 현재 실무의 문제 | FairValue Engine의 해결 |
|---|---|
| CB/RCPS 평가가 담당자별 엑셀에 산재 → 재현 불가, 검증 어려움 | 입력·모형·결과를 서버에 스냅샷 저장, 동일 입력 → 동일 결과 보장 |
| 리픽싱·희석·상환·매도청구권 등 복잡 조건이 수식에 하드코딩 | 권리조건을 데이터로 모델링, 엔진이 조건 조합을 일관 처리 |
| 신용등급 커브를 매번 수기 입력 | 기준일·등급별 커브 자동 매핑 + 업로드 이력 관리 |
| 감사 대응 시 "계산근거" 재작성에 시간 소요 | 현금흐름·할인계수·노드/패스 등 계산검증 데이터 자동 보관·내보내기 |
| 비상장 기초자산 변동성 산출이 비표준적 | 유사기업군 기반 변동성 산출 로직 내장 |
| IPO/다운라운드/조기상환 등 What-if 분석이 수작업 | 시나리오·민감도 분석 자동화 |

**시장 배경**: 파생상품·메자닌의 복잡도 증가와 K-IFRS 공정가치 측정 의무로, 정확하고 *검증 가능한* 공정가치 산출 수요가 구조적으로 존재한다. 본 플랫폼은 평가 정확성보다 **재현성·추적성·문서화**를 1차 차별점으로 삼는다(정확성은 모형의 정합성으로, 신뢰는 감사 추적성으로 확보).

| 스타트업 CFO/재무담당 | 투자유치(RCPS/CB) 후속 공정가치 | RCPS/CB Pricing, 시나리오(IPO/다운라운드) |

## 5. 핵심 가치 제안 (Value Proposition)

- **One platform, all 메자닌**: RCPS·CPS·CB·EB·BW·SO를 단일 엔진 라우팅으로 처리.
- **Reproducible by design**: 입력 스냅샷 해시 + 모델 버전 + 시드 고정 → 감사·재계산 100% 재현.
- **Audit-ready output**: 결과파일(Excel) + 보고서(PDF) + 계산근거 패키지(zip) 자동 생성.
- **Domain-correct validation**: 단순 null 체크가 아니라 *금융적으로* 부정합한 입력(예: parity·할인율·날짜·권리조건 충돌)을 차단.
- **Curve governance**: 기준일·등급별 커브 버전 관리, 수기 커브 vs Bootstrapping 커브 우선순위 규칙.

## 6. 전체 서비스 범위 (Scope)

**In-scope (v1)**
- 7개 상품 평가엔진(RCPS, CPS, CB, EB, BW, SO, 조건부 SO)
- 등급커브 업로드/버전관리/수기입력 + Bootstrapping(Zero/Forward/DF 생성)
- 평가 워크플로우(Wizard), 계산 전 검증, 비동기 계산(Job)
- 결과 대시보드, 민감도·시나리오 분석
- 결과파일(Excel)·보고서(PDF)·계산근거 패키지
- 평가 이력/스냅샷, 임시저장, 링크 공유
- 조직·사용자·역할 권한, Audit Trail(계산근거·접근 로그)

**Out-of-scope (v1)**
- 내부통제/MRC 워크플로우(검토·승인 결재선) — 명시적 제외
- 실시간 시장데이터 자동 수신(외부 단말 직접 연동) — v1은 업로드/수기 + 정적 마스터
- 다통화 IRS/CRS/통화선도 등 순수 장외파생(로드맵 §26에 확장 항목으로)

## 7. 지원 상품군 정의

각 상품을 **개요 / 평가목적 / 핵심 계약조건 / 주요 입력값 / 권리조건 / 시장 파라미터 / 적용 모형 / 결과 항목 / 검증 규칙 / 보고서 항목 / 공통 모듈 / 특수 모듈**로 정의한다.

### 7.1 RCPS — 상환전환우선주

- **개요**: 우선주 + 상환권(보유자 풋 성격, 보장수익률로 상환) + 전환권(보통주 전환). 리픽싱·희석 빈번.
- **평가목적**: K-IFRS상 내재파생 분리 평가 및 전체 공정가치(투자자/발행자 관점) 산출.
- **핵심 계약조건**: 발행일·만기일, 발행금액·발행가, 우선배당률, 보장수익률, 전환조건, 상환조건, 리픽싱 조건.
- **주요 입력값**: 발행일, 만기일, 발행금액, 주당 발행가액, 우선배당률(%), 지급주기(M), 보장수익률(%).
- **권리조건**: 전환권(필수), 상환권(필수), 매도청구권(선택), 리픽싱(선택).
- **시장 파라미터**: 무위험커브, 신용등급커브, 기초주가, 변동성, 배당률, 기발행 주식수.
- **적용 모형**: 이항/삼항 격자(보유·전환·상환 동시 최적화) 기본, LSMC(리픽싱 경로의존) 보조, DCF(상환 floor).
- **결과 항목**: 우선주 가치, 전환권 가치, 상환권 가치, 매도청구권 가치, 총 공정가치, Parity, 민감도.
- **검증 규칙**: 보장수익률 ≥ 0, 전환가 > 0, 상환행사기간 ⊆ [발행일, 만기일], 격자 vs MC 오차 한계.
- **보고서 항목**: 계약조건, 적용커브, 모형·가정, 구성요소별 가치, 민감도/시나리오.
- **공통 모듈**: 커브 매핑, 기초자산 매핑, 변동성 산출, 격자/MC 엔진, 리픽싱 엔진, 희석 조정.
- **특수 모듈**: 우선배당 누적/비누적 처리, 상환수익률 누적 보장(IRR target) 로직.

### 7.2 CPS — 전환우선주

- **개요**: RCPS에서 **상환권 제거**. 우선주 + 전환권(+리픽싱·희석).
- **평가목적**: 전환옵션 내재 우선주의 공정가치 산출.
- **핵심 계약조건**: RCPS와 동일하되 상환 조건 없음.
- **권리조건**: 전환권(필수), 리픽싱(선택). 상환권/매도청구권 비활성.
- **적용 모형**: 이항 격자 또는 (BS 전환옵션 + 우선주 host PV), MC(리픽싱).
- **결과 항목**: 우선주 가치, 전환권 가치, 총 공정가치, Parity.
- **특수 모듈**: 영구·비만기형 우선주의 host 가치 산정(배당 영구연금 근사) 옵션.

### 7.3 CB — 전환사채

- **개요**: 일반사채 + 전환권 + 발행자 조기상환권(콜) + 투자자 조기상환청구권(풋) + 매도청구권(제3자/발행자 콜) + 리픽싱.
- **평가목적**: 채권요소·지분요소 분리(이중 할인) + 내재옵션 동시 평가.
- **핵심 계약조건**: 표면이자율, 지급주기, 만기, 보장수익률(YTP/YTM 보장), 전환가, 콜/풋 일정·가격, 매도청구권 조건.
- **권리조건**: 전환권(필수), 상환권(콜/풋, 선택), 매도청구권(선택), 리픽싱(선택).
- **시장 파라미터**: 무위험커브, 신용스프레드(등급커브), 주가, 변동성, 배당률.
- **적용 모형**: **Tsiveriotis–Fernandes 격자(기본)**, LSMC(리픽싱·경로의존 보조), MC, DCF(bond floor), 매도청구권은 독립파생 분리평가 옵션.
- **결과 항목**: 채권가치, 전환권 가치, 상환권 가치, 매도청구권 가치(부호 주의), 총 공정가치, Risk Free Rate, YTM, Parity.
- **검증 규칙**: 총가치 ≥ bond floor, 전환권 ≥ max(parity−bond,0) 근방, 콜 ≤ 0 영향(보유자 관점), 풋 ≥ 0 영향.
- **공통 모듈**: 커브, 기초자산, 변동성, 격자/MC, 리픽싱, 신용할인.
- **특수 모듈**: equity/debt 컴포넌트 이중 할인(T-F), 매도청구권 독립파생 분리 평가.

### 7.4 EB — 교환사채

- **개요**: 일반사채 + **타사 주식으로 교환**하는 옵션(기존 주식 → 발행자 희석 없음).
- **평가목적**: 채권가치 + 교환옵션 분리 평가(신용은 EB 발행자, 옵션 기초는 대상사 주식).
- **권리조건**: 교환권(필수), 상환권(선택). 희석효과 미적용(기존주식).
- **적용 모형**: DCF bond floor + BS/격자 교환옵션, MC(대상주 경로).
- **결과 항목**: 채권가치, 교환권 가치, 총 공정가치.
- **특수 모듈**: 교환 대상사 주가·변동성·배당 별도 매핑(발행사와 다른 종목).

### 7.5 BW — 신주인수권부사채

- **개요**: 일반사채 + 신주인수권(워런트, 신주 발행 → **희석 적용**). 분리형/비분리형.
- **평가목적**: 사채가치 + 워런트(희석조정) 분리 평가.
- **권리조건**: 신주인수권(필수), 상환권(선택), 희석효과(적용).
- **적용 모형**: DCF 사채 + BS 워런트(희석조정), 격자(미국형 워런트).
- **결과 항목**: 채권가치, 신주인수권 가치, 총 공정가치, 희석효과.
- **특수 모듈**: 희석계수 `N/(N+M)` 적용(N=기발행, M=워런트 신주), 분리형 거래가치 분해.

### 7.6 Stock Option — 스톡옵션 (주식기준보상)

- **개요**: 임직원 부여 주식매수선택권. K-IFRS 1102에 따라 부여일 공정가치를 가득기간에 비용 인식.
- **평가목적**: 부여일 공정가치(grant-date fair value) + 비용 인식 스케줄.
- **주요 입력값**: 부여일, 평가기준일, 만기일, 행사가, 기초주가, 기대변동성, 기대배당률, 무위험이자율, 기대만기, 퇴사율, 가득조건, 행사패턴, 조기행사 가정.
- **적용 모형**: BSM(조정), 이항 격자(조기행사 multiple), MC.
- **결과 항목**: 옵션 단가, 총 옵션가치, 보상원가, 가득기간별 비용 인식 스케줄.
- **특수 모듈**: 기대만기 추정, 조기행사 배수(suboptimal exercise factor), 퇴사율 반영 수량 조정.

### 7.7 Conditional Stock Option — 조건부 스톡옵션

- **개요**: 시장조건(목표주가/TSR)·비시장조건(근무·성과: 매출/EBITDA)·가득조건이 결합된 주식보상.
- **회계 핵심**: **시장조건은 공정가치에 내재**(미충족 시 비용 환입 없음, MC로 반영), **비시장조건은 가득 *수량* 으로 조정**(true-up).
- **적용 모형**: MC(시장조건), LSMC(경로/조기행사), Barrier(목표주가 KI/KO), 확률조정(비시장조건), Vesting condition model.
- **결과 항목**: 조건부 옵션 가치, 조건 충족 확률, 보상원가, 조건별 비용 인식 스케줄, 시나리오별 가치.
- **특수 모듈**: 시장/비시장 조건 분리 처리기, 성과지표 경로 시뮬레이터, 확률 가중 비용 스케줄러.

## 8. 상품별 평가모형 설계 (Pricing Model Design)

> 본 절은 이 문서의 **최심부**다. 각 상품의 수리적 처리와 계산 난점, 검증법을 정의한다.

### 8.1 공통 평가 프레임워크 (Risk-Neutral / No-Arbitrage)

모든 옵션성 평가는 위험중립 측도 하에서 기대 페이오프를 무위험할인한 값으로 정의한다.

- **할인 이원화(T-F 핵심)**: 지분요소(전환·교환·워런트 등 주식으로 귀결되는 흐름)는 **무위험커브**로, 부채요소(쿠폰·원금·상환 등 현금으로 귀결되는 흐름)는 **무위험 + 신용스프레드(등급커브)** 로 할인한다.
- **격자(Lattice)**: 기초주가에 CRR(Cox–Ross–Rubinstein) 이항 또는 삼항 트리를 적용. 노드별 backward induction에서 보유/전환/상환/콜/풋 중 최적가치 선택.
- **시뮬레이션(MC/LSMC)**: 경로의존(리픽싱, 시장조건, 배리어) 또는 다중 권리 상호작용 시 사용. 미국형/버뮤다형 조기행사는 **LSMC(Longstaff–Schwartz)** 회귀로 행사경계 추정.
- **재현성**: MC는 시드 고정 + 안티테틱/소볼(저불일치수열) 분산축소. 동일 입력 → 동일 결과를 계약상 보장.

### 8.3 핵심 계산 로직 상세

**(1) CB — Tsiveriotis–Fernandes 격자 (의사 로직)**

```
입력: S0(주가), σ(변동성), q(배당), 무위험커브 r(t), 신용스프레드 cs(t),
      쿠폰스케줄 C, 액면 F, 전환비율 k, 콜/풋/매도청구권 일정
1. 트리 구성: up u=exp(σ√Δt), down d=1/u, p=(exp((r−q)Δt)−d)/(u−d)
2. 만기 노드: 보유가치 V=F+마지막쿠폰; 전환가치 CV=k·S
   - 부채부 B_D = (V가 채권상환으로 실현되면) F, 지분부 B_E = (전환되면) CV
3. backward induction (노드별):
   hold_E = e^{−rΔt}·E[B_E];  hold_D = e^{−(r+cs)Δt}·E[B_D]   # 지분부=무위험, 부채부=위험할인
   continuation = hold_E + hold_D + 해당기간 쿠폰
   if 전환가능: convert = k·S  → 전환 시 전부 지분부로 재분류
   if 발행자 콜 가능: 발행자는 min(continuation, max(call_price, convert)) 강제
   if 투자자 풋 가능: 투자자는 max(continuation, put_price)
   노드값 = 위 규칙의 최적 선택, 선택 결과에 따라 (B_E,B_D) 재배분
4. 리픽싱: 전환가 K가 경로상 하향조정 → 격자만으로 부정확 시 LSMC로 전환
5. 매도청구권(독립파생): 별도 트리/closed-form으로 평가 후 부호 반영(보유자 −)
출력: 채권가치=Σ부채부 기여, 전환권가치=총가치−채권−상환, 상환권=풋/콜 기여 분해, Parity=S/K×100
```

**(2) RCPS — 격자 + 상환 IRR 보장**
- 상환가치는 보장수익률 기반 **누적 보장 원리금**(예: 발행가×(1+보장수익률)^t)을 풋 행사가로 사용.
- 노드별: `max(전환가치, 보유가치, 상환행사가치)`, 발행자 콜이 있으면 콜 제약 추가.
- 우선배당 누적형은 미지급 배당을 상환가치에 가산.

**(3) 리픽싱 엔진 (down-round 보호)**
- 입력: Refixing StartDate, StepMonth(재조정 주기), Refixing Floor(하한), 최초 행사가액, 조정방식(상/하향).
- 로직: 재조정 시점마다 `K_new = max(Floor, min(K_old, 시가기준조정가))`. 경로의존이므로 MC에서 각 패스의 K 궤적을 추적.
- 검증: K_new ∈ [Floor, 최초행사가], 단조 하향(하향 리픽싱) 또는 양방향(상-하향) 규칙 일관성.

**(4) 희석 조정 (BW/전환류)**
- 워런트/전환으로 신주 M주 발행, 기발행 N주 → 옵션가치에 `N/(N+M)` 희석계수. 더 정밀히는 기업가치(부채+자기자본) 기반 희석 모형.
- EB는 기존주식 교환이므로 희석 미적용(0).

**(5) Stock Option (K-IFRS 1102)**
- BSM: `c = S·e^{−qT}·N(d1) − X·e^{−rT}·N(d2)`, T=기대만기. 퇴사율은 부여수량×(1−forfeiture)로 수량 조정.
- 비용 인식: 총 보상원가 = 단가 × 가득예상수량, 가득기간에 정액/가속 배분.

**(6) 조건부 SO (시장 vs 비시장 분리)**
- **시장조건**(목표주가): MC 경로에서 조건 충족 시에만 페이오프 → 부여일 공정가치에 *내재*. 미충족이어도 비용 환입 없음.
- **비시장조건**(매출/EBITDA·근무): 공정가치에는 미반영, 대신 **가득 예상 수량**을 충족확률로 추정하여 비용에 반영(매 보고일 true-up).
- 배리어(KI/KO) 목표주가 구조는 배리어 옵션 모형 병행.

### 8.4 모형 선택 기본값(Default) 규칙

| 조건 | 기본 모형 |
|---|---|
| 단순 전환(리픽싱 없음, 단일 기초) | 이항 격자(CB는 T-F 격자) |
| 리픽싱 있음(경로의존) | LSMC |
| 다중 기초/배리어/시장조건 | Monte Carlo |
| 미국형/버뮤다 조기행사 | LSMC |
| 채권 floor·단순 우선주 host | DCF |
| 표준 ESO(조기행사·퇴사 단순) | BSM, 복잡 시 이항 격자 |

사용자는 기본값을 override 할 수 있으며, **선택 모형과 입력 조건의 정합성**(예: 리픽싱 있는데 단순 BS 선택)은 §19 검증에서 경고한다.

## 9. Pricing Engine 아키텍처

평가엔진은 **독립 서비스(Python)** 로 분리한다. 업무 API(상태·CRUD·권한)는 Kotlin/Spring, 계산은 Python(QuantLib/NumPy/SciPy)으로 책임을 나눈다.

### 9.1 엔진 내부 구조

```
[Pricing Engine (Python/FastAPI)]
 ├─ Router        : 상품코드 → 평가 핸들러 매핑 (RCPS/CPS/CB/EB/BW/SO/CSO)
 ├─ InputResolver : 입력 + 매핑된 커브/시장데이터 → 정규화된 ValuationContext
 ├─ CurveService  : zero/forward/DF 빌드, 보간, 신용스프레드 가산
 ├─ ModelLibrary  : Lattice(CRR), TF-Lattice, MC/LSMC, BSM, Barrier, DCF
 ├─ RightsEngine  : 전환/교환/신주인수/상환(콜·풋)/매도청구권/리픽싱/희석 처리
 ├─ Calculator    : backward induction / path simulation 실행
 ├─ ResultBuilder : 구성요소 분해, 파라미터(RFR/YTM/Parity), 민감도/시나리오
 ├─ AuditWriter   : 현금흐름·할인계수·노드/패스·행사판단 로그 직렬화
 └─ Reproducer    : 시드/버전/입력해시 고정, 재계산 동일성 보장
```

### 9.2 계산 요청 처리 흐름 (비동기 Job)

| 단계 | 내용 |
|---|---|
| 1. 요청 | Backend가 정규화 입력 + 커브 스냅샷을 Job Queue에 enqueue |
| 2. 큐잉 | `pricing_jobs` 레코드 생성(status=QUEUED), job_id 반환 |
| 3. 실행 | Worker가 Engine 호출(RUNNING), MC/LSMC는 멀티프로세스 분산 |
| 4. 저장 | 결과(`pricing_results`) + 중간산출(`simulation_results`) 저장, status=DONE |
| 5. 폴링/푸시 | FE가 상태 폴링 또는 WebSocket 알림 수신 후 결과 렌더 |

> 단순 격자/BSM은 동기(수백 ms) 처리도 허용하되, **모든 결과는 Job 스냅샷으로 영속화**해 재현성·이력을 통일한다.

### 9.3 단일 백엔드 vs 마이크로서비스 (의사결정)

| 구조 | 장점 | 단점 | 결론 |
|---|---|---|---|
| 단일(모놀리식, JVM에 엔진 포함) | 단순 배포, 호출 지연 적음 | 금융계산을 JVM에 재구현(QuantLib 미활용), 스케일 분리 불가 | 부적합 |
| **분리(Spring + Python Engine)** | QuantLib/NumPy 활용, 계산만 독립 스케일·버전관리 | 서비스 간 통신·계약 관리 비용 | **채택(v1)** |
| 풀 마이크로서비스(상품별 분리) | 상품별 독립 배포 | 초기 운영 과다, v1 과설계 | v2 이후 검토 |

v1은 **2-서비스(Backend / Pricing Engine)** + 공용 PostgreSQL/Redis/Queue로 시작하고, 엔진만 수평 확장한다.

## 10. Parameter 및 등급 커브 관리

### 10.1 커브 데이터 모델

- **무위험등급커브**: 기준일·만기(Term)별 무위험 수익률.
- **신용등급커브**: 기준일·신용등급·만기별 수익률(또는 무위험+스프레드).
- 만기 그리드(고정 컬럼): `3M, 6M, 9M, 1Y, 1.5Y, 2Y, 3Y, 4Y, 5Y, 7Y, 10Y, 15Y, 20Y, 30Y, 50Y`.

### 10.2 업로드/검증 규칙

| 검증 | 규칙 |
|---|---|
| 파일 형식 | Excel/CSV, 지정 스키마(원본 파일 그대로 업로드, 가공 금지) |
| 필수 컬럼 | 기준일·종류·구분·적용신용등급 + 만기 그리드 전체 |
| 값 범위 | Rate 수치형, 비정상치(음수 과대, 1000bp 초과 급변) 경고 |
| 단조성 | 만기 증가에 따른 비정상 역전 경고(곧바로 차단은 아님) |
| 중복 | 동일 (기준일, 등급) 재업로드 시 버전 증가 또는 덮어쓰기 선택 |

### 10.3 자동 매핑 & 폴백

- 평가기준일 + 적용신용등급 → 해당 (기준일, 등급) 커브 자동 매핑.
- **폴백**: 정확한 기준일 커브가 없으면 "가장 가까운 이전 기준일" 사용 여부를 사용자가 선택(기본 OFF, 선택 시 보고서에 명시).
- **수기 입력**: 등급커브를 사용자가 직접 입력(감사 추적에 "수기" 플래그).
- **보간**: 평가에 필요한 비표준 만기는 커브 보간으로 산출(§11).

### 10.4 우선순위 규칙 (수기 vs 업로드 vs Bootstrapping)

```
1순위: 사용자가 명시적으로 "수기 입력"한 커브 (해당 평가 한정)
2순위: 조직이 업로드한 (기준일, 등급) 커브
3순위: Bootstrapping으로 생성된 Zero/DF 커브 (업로드 par-rate로부터 파생)
폴백 : 가장 가까운 이전 기준일 커브 (사용자 동의 시)
```
→ 동일 평가 내 우선순위·출처·기준일을 결과/보고서에 항상 기록한다.

## 11. Bootstrapping 기능 설계

| 항목 | 정의 |
|---|---|
| **목적** | 시장 par/현물 금리에서 무차익 **Zero(현물)커브·Discount Factor·Forward** 산출 → 모든 할인의 단일 소스 |
| **입력 시장금리** | 업로드된 등급커브의 만기별 par yield(또는 스왑/국고 금리) |
| **Zero Curve 생성** | 단기는 직접, 장기는 순차 부트스트랩: `DF(t_n)`을 par 조건식에서 역산 |
| **Discount Factor** | `DF(t)=1/(1+z(t))^t` (이산) 또는 `exp(−z(t)·t)` (연속). 내부는 연속복리 통일 |
| **Forward Rate** | `f(t1,t2)=(DF(t1)/DF(t2)−1)/(t2−t1)` (단순) 또는 `ln(DF(t1)/DF(t2))/(t2−t1)` (연속) |
| **보간 방식** | 기본=zero rate 선형 / 옵션=log-linear on DF, monotone-convex, cubic spline. 음의 forward 방지 위해 monotone-convex 권장 |
| **커브 저장** | (기준일, 등급, 보간법, 생성시각, 입력 par 스냅샷) 단위로 버전 저장 |
| **Engine 연동** | Engine은 항상 DF(t)/z(t) 인터페이스로 조회 → 모형은 커브 생성법에 비의존 |
| **우선순위** | §10.4 규칙 적용(수기 > 업로드 par > bootstrap 파생) |

검증: 부트스트랩 후 입력 par 상품을 재평가 시 **par(=100)로 복원되는지** 자기검산(round-trip), 음의 forward 경고.

### 12.1 Pricing Wizard (7단계, 재설계)

```
Step 1 상품 선택        → 상품 카드 선택, 해당 상품의 폼 스키마 동적 로드

Step 7 계산·결과        → 비동기 계산 → 결과 대시보드 → 민감도/시나리오 → 보고서/파일
```

좌측 **Summary/검증 패널**이 전 단계에서 상품·기준일·입력 완성도(%)·검증 상태를 고정 표시한다. 단계 자유 이동 가능(검증은 계산 직전 일괄 + 단계별 인라인 병행).

## 17. 평가 결과 데이터 설계

### 17.1 결과 표준 스키마

| 그룹 | 항목 |
|---|---|
| Summary | 총 공정가치, 단위당(주당/액면당) 가치, 평가손익(장부가 대비) |
| Component | 채권/우선주(host), 전환권, 교환권, 신주인수권, 상환권, 매도청구권(부호 포함) |
| Key Parameters | Risk Free Rate, YTM, Credit Spread, Volatility, Dividend Yield, Parity |
| Reproducibility | 모델명·버전, 시드, 입력 스냅샷 해시, 커브 출처·기준일, 계산시각·소요 |

### 17.2 구성요소 부호 규약

- 보유자(투자자) 관점 기본. **발행자/제3자가 보유한 콜(매도청구권·발행자 조기상환콜)** 은 보유자 가치를 **차감(−)**, 투자자 풋·전환권은 **가산(+)**.
- 총 공정가치 = host(채권/우선주) + Σ(보유자 유리 옵션) − Σ(발행자/제3자 유리 옵션).

### 17.3 Parity 정의

`Parity(%) = 주가 ÷ 전환(행사)가격 × 100`. 표시 스케일(액면 10,000 기준 환산 등) 규약은 상품 설정에 명시하고 보고서에 표기.

## 19. 검증 및 품질관리 체계

### 19.1 계산 전 입력 검증(인라인 + 일괄)

| 분류 | 검증 항목 |
|---|---|
| 필수값 | 누락 항목 차단, 항목 딥링크 |

| 수치 | 음수 주가/변동성, 비정상 할인율(과대/과소), 희석계수 범위 |
| 데이터 | 커브 누락, 신용등급 미선택, 기초자산 미매핑 |
| 정합성 | 상품조건↔모형 부정합(리픽싱 있는데 단순 BS), 할인유형 미선택 |

검증 결과는 **오류(차단)/경고(허용+주석)** 2단계. 오류 0건이어야 "계산" 활성.

### 19.2 계산 후 결과 검증

| 점검 | 기준 |
|---|---|
| 결과 범위 | 음수 가치 발생 여부(허용 부호 외) |
| 모형 교차검증 | DCF vs Tree vs MC 허용오차 내 일치 |
| 하한 조건 | 총가치 ≥ bond floor, 옵션 ≥ intrinsic |
| Parity | 비정상 Parity 경고 |
| MC 수렴 | 표준오차/신뢰구간 한계, 패스 수 충분성 |
| 재현성 | 동일 입력 재계산 결과 일치(해시 비교) |

### 19.3 계산검증 데이터(Audit 산출물)

- 현금흐름 스케줄, 할인계수(DF) 테이블, Tree Node 요약, Simulation Path 요약, 권리행사 판단 로그, 모델별 중간 산출값, 최종 가치 합산 테이블.

- 발행정보·결과데이터를 구조화하여 후속 결산 시스템 연계 용이.

### 20.3 Audit Pack(zip)

`입력 스냅샷(JSON) + 적용커브(CSV) + 계산검증 데이터 + 결과(JSON) + 보고서(PDF) + 메타(모델버전·시드·해시)` 를 단일 패키지로 내보내 감사 대응.

## 21. 데이터베이스 설계

PostgreSQL 기준. 핵심 원칙: **입력·커브·결과를 스냅샷으로 영속화**(재현성), 조직(테넌트) 격리, 감사 로그 분리.

|---|---|---|---|---|
| `users` | 사용자 | id, org_id, email, name, role, status | PK id / FK org_id→organizations | N:1 org |
| `organizations` | 조직(테넌트) | id, name, plan, created_at | PK id | 1:N users/curves |
| `projects` | 평가 묶음(선택) | id, org_id, name, owner_id | PK id / FK org_id, owner_id | 1:N instruments |
| `instruments` | 평가 대상 상품 | id, org_id, project_id, type(RCPS..), issuer, name, status | PK id / FK org_id, project_id | 1:1 terms, 1:N jobs |
| `instrument_terms` | 상품 기본 계약조건 | instrument_id, issue_date, maturity_date, face, issue_price, coupon_rate, freq_m, guaranteed_yield | PK id / FK instrument_id | 1:1 instrument |
| `option_terms` | 전환/교환/신주인수 조건 | id, instrument_id, kind, conv_price, ratio, start, end, dilution_flag, underlying_id | FK instrument_id, underlying_id | N:1 instrument |

| `dividend_data` | 배당 | id, asset_id, as_of, div_yield, source | FK asset_id | N:1 asset |

| `pricing_parameters` | 평가에 적용된 파라미터 스냅샷 | id, job_id, rfr, ytm, credit_spread, vol, div, parity, curve_source | FK job_id | 1:1 job |

## 29. 개발자용 요구사항 요약 (Backend/Engine)

- **Backend(Spring)**: 인증/RBAC(조직격리), 상품/권리조건 CRUD, Job 오케스트레이션(enqueue·상태·결과 영속화), 입력 정규화·해시(SHA-256), 스냅샷/이력/링크, 파일·보고서 발급, audit_logs.
- **Engine(Python)**: 상품 라우터, ValuationContext 정규화, CurveService(zero/df/forward·보간·신용가산), ModelLibrary(CRR/T-F/MC/LSMC/BSM/Barrier/DCF), RightsEngine(전환/교환/신주인수/콜·풋/매도청구권/리픽싱/희석), AuditWriter(현금흐름·DF·노드·패스·행사로그), Reproducer(시드·버전·해시).
- **계약(Contract)**: Backend↔Engine은 정규화 JSON(입력+커브 스냅샷)으로 통신, Engine은 stateless. 동일 input_hash → 캐시 결과.
- **테스트**: 모형별 골든값 회귀, 격자 vs MC 허용오차, round-trip(par 복원), 재현성(동일입력 동일결과) E2E.

## 30. 디자이너용 화면 설계 요약

- **톤**: 무채색 베이스 + 단일 강조색, 그라데이션·과한 카드·허수 위젯 배제. 데이터그리드·정렬·고정헤더 중심의 "재무 실무 그레이드".
- **레이아웃**: Stepper + 좌측 Summary/검증 패널(완성도·검증·상품·기준일 상시), 입력과 결과 화면 분리.
- **동적 폼**: 상품별로 존재하는 권리조건만 노출(전환/상환/매도청구권/리픽싱 토글), 비활성 조건 숨김.
- **검증 UX**: 항목 옆 인라인 경고 + "수정하러 가기" 딥링크(오류=차단/경고=주석 2단계).
- **결과 대시보드**: Summary 카드 → Component breakdown(부호·색) → Key Parameters → 민감도(토네이도)·시나리오(비교표) → 계산로그/다운로드.
- **숫자 표기**: 통화·% 우측정렬·천단위·음수 색+괄호, 단위 라벨 명시. 재현 메타(모델버전·시드·커브출처) 표시 영역.
- **독립 재설계 의무**: 벤치마킹 화면의 배치·색·문구·버튼 순서를 복제하지 않는다(§14.3).

## 31. 데이터/평가엔진 담당자용 요구사항 요약

- **커브 파이프라인**: 업로드 스키마(만기 3M~50Y 고정) 검증 → 등급별 버전 저장 → (기준일,등급) 자동 매핑 → 비표준 만기 보간 → Bootstrapping(zero/df/forward, monotone-convex 권장) → 우선순위 규칙(수기>업로드>부트스트랩, 폴백=최근 기준일).
- **시장데이터 마스터**: 기초자산(상장/비상장), 변동성(출처·기준일·산출법), 배당. 비상장 변동성=유사기업 합성(가중/중앙값+비유동성 조정), 수기 허용·플래그.
- **모형 정합성**: 상품-모형 매핑(§8.2)·기본값 규칙(§8.4) 준수, 조건↔모형 부정합 검증.
- **검증·재현성**: 모형 교차검증·round-trip·MC 표준오차 기준 수립, 시드·버전·input_hash로 동일성 보장, 계산검증 데이터(현금흐름·DF·노드·패스·행사로그) 직렬화·보관.
- **회계 정합**: K-IFRS 1102 시장조건(FV 내재)·비시장조건(수량 true-up) 분리, CB equity/debt 이중할인, 매도청구권 독립파생 분리(할인유형) 처리.
